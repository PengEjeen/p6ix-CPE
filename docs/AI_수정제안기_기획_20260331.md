## 1. 목적과 배경

### 1.1 목적
본 문서는 공기적정성 검토 시스템에 `구글 Docs 제안 모드`와 유사한 승인형 편집 경험을 제공하는 `AI 수정 제안기` 도입 방향을 정리한다. 핵심 목표는 AI가 공정표를 직접 확정 수정하는 것이 아니라, 사용자가 우측 패널에서 AI와 대화하며 수정안을 검토하고 `사용 여부를 결정`하도록 지원하는 것이다.

### 1.2 배경
현재 시스템은 공사기간 산정용 일정표를 테이블/간트 기반으로 편집하며, 일부 AI 보조 기능은 일정 단축 요약 수준으로 존재한다. 그러나 실무 검토에서는 다음 요구가 지속적으로 발생할 수 있다.

- 비현실적인 생산성, 인력, 병행률 입력의 빠른 탐지
- 목표 공기 기준의 조정안 제시
- 제안 이유와 근거의 설명 가능성 확보
- 자동 반영이 아닌 사용자 승인 기반의 안전한 편집 경험
- 동일 화면 안에서 AI와 후속 대화를 이어가며 수정안을 다듬는 경험

### 1.3 문서 해석 원칙

- `사실(Fact)`: 현재 코드와 화면에서 확인 가능한 내용
- `제안(Proposal)`: 본 문서에서 도입을 제안하는 기능 방향
- `오픈 이슈(Open Issue)`: 정책 또는 상세 설계가 추가로 필요한 항목

## 2. 기능 정의

### 2.1 기능 한 줄 정의
`AI 수정 제안기`는 일정 마스터 화면의 우측 패널에서 사용자가 자연어로 수정 요청을 입력하면, AI가 `변경 제안 카드`를 생성하고 사용자가 카드별 `사용`, `거절`, `다시 제안`을 결정하도록 지원하는 대화형 검토 보조 기능이다.

### 2.2 기존 기능과의 차이

사실(Fact):

- 현재 일정표는 테이블과 간트에서 직접 편집된다.
- 현재 AI 보조는 요약 및 일정 단축 계산 보조 성격이 강하다.
- 현재 AI UI는 로그/요약/일괄 적용 중심에 가깝다.

제안(Proposal):

- 신규 기능은 일반적인 범용 잡담형 챗봇이 아니라 `공정표 수정 제안`에 특화된 대화형 패널로 설계한다.
- AI 응답은 자연어 답변만 제공하지 않고 `적용 가능한 제안 카드`를 함께 반환한다.
- 사용자는 패널에서 후속 질문을 이어가며 제안 범위를 좁히거나 다른 안을 다시 받을 수 있어야 한다.

## 3. 핵심 원칙

### 3.1 동작 원칙

제안(Proposal):

- AI는 제안만 하고 직접 저장하지 않는다.
- 모든 제안은 `변경 전`, `변경 후`, `제안 사유`, `영향 범위`를 함께 보여준다.
- 사용자는 제안별 `사용`, `거절`, `다시 제안`을 선택할 수 있어야 한다.
- 여러 제안이 있을 경우 `선택 적용`, `전체 적용`, `모두 거절`이 가능해야 한다.
- 수락된 제안만 현재 화면 데이터에 반영된다.
- 최종 영속 저장은 기존 저장 로직이 수행한다.
- 제안 결과와 사용자 선택 이력은 대화 스레드 기준으로 추적 가능해야 한다.

### 3.2 신뢰성 원칙

제안(Proposal):

- AI는 도메인 계산을 대체하지 않는다.
- 공기 계산, 링크 정합성, 일정 재산정은 기존 시스템 로직이 책임진다.
- AI는 `설명`, `우선순위화`, `수정 후보 제시`, `비교안 생성` 역할에 집중한다.
- 제안 적용 직후에도 기존 검증 및 재계산 로직이 다시 실행되어야 한다.

### 3.3 UX 원칙

제안(Proposal):

- 사용자는 화면을 벗어나지 않고 대화, 비교, 승인, 반영을 끝낼 수 있어야 한다.
- 패널의 대화 메시지와 실제 데이터 변경 카드는 시각적으로 구분되어야 한다.
- 승인되지 않은 제안은 원본 데이터를 덮어쓰지 않아야 한다.
- 제안이 있는 셀/행은 테이블과 간트에서 하이라이트되어야 한다.
- 사용자는 AI 제안을 `참고안`으로 인식해야 하며, 시스템은 이를 유도하는 문구를 유지해야 한다.

## 4. 사용자 시나리오

### 4.1 기본 시나리오

1. 사용자가 일정표에서 특정 행, 대공종, 또는 현재 필터 범위를 선택한다.
2. 사용자가 우측 `AI 제안 패널`을 연다.
3. 사용자가 예: `이 구간 목표 공기를 10일 줄이고 싶어. 무리한 인력 증원은 피해서 제안해줘.` 같은 요청을 입력한다.
4. 시스템이 선택 범위와 현재 일정 데이터를 기준으로 AI 응답을 생성한다.
5. 패널에는 AI 설명 메시지와 함께 `수정 제안 카드`가 표시된다.
6. 사용자가 카드별로 `사용`, `거절`, `다른 안 보기`를 선택한다.
7. `사용`한 카드만 화면 데이터에 반영된다.
8. 사용자는 필요 시 후속 질문으로 추가 조정을 요청한다.
9. 사용자는 기존 저장 또는 되돌리기 흐름을 그대로 사용한다.

### 4.2 후속 대화 시나리오

제안(Proposal):

- 사용자는 `이 행만 다시 제안해줘`, `생산성 말고 인력만 조정해줘`, `방금 제안보다 보수적으로 다시`처럼 후속 요청을 보낼 수 있어야 한다.
- AI는 직전 대화와 사용자 선택 상태를 반영해 새 제안을 생성해야 한다.
- 이미 거절된 제안과 동일한 안을 반복 제시할 경우 이유 없이 재노출하지 않도록 제어해야 한다.

### 4.3 기대 사용자 경험

제안(Proposal):

- 사용자는 패널을 `대화창`으로 인식하되, 실제 핵심 산출물은 `제안 카드`라고 받아들여야 한다.
- 사용자는 AI가 무엇을 바꾸려는지 한 눈에 보고, 채택 여부를 빠르게 결정할 수 있어야 한다.
- 제안을 적용한 뒤에도 원래 편집 흐름과 충돌하지 않아야 한다.

## 5. 제안 대상과 단위

### 5.1 제안 대상

제안(Proposal):

- 선택한 세부공정 행
- 선택한 대공종 전체
- 검색/필터 결과에 포함된 현재 보이는 항목
- 사용자가 패널에서 명시적으로 지정한 범위

초기 범위에서는 전체 프로젝트 일괄 제안보다 `사용자가 명시적으로 선택한 범위`를 우선한다.

### 5.2 제안 단위

제안(Proposal):

- 공정표 문장 단위가 아니라 `행/필드 단위` 제안
- 필요 시 하나의 카드가 복수 필드 변경을 포함할 수 있음
- 예시 필드:
  - `crew_size`
  - `productivity`
  - `parallel_rate`
  - `reflection_rate`
  - `note`

예시 표현:

- `토공사 > 굴착 공정의 crew_size를 2에서 3으로 제안`
- `골조공사 > 슬래브 공정의 productivity를 100에서 110으로 제안`
- `마감공사 비고 문구를 설명형으로 정리 제안`

## 6. 화면 기획 방향

### 6.1 화면 구조

제안(Proposal):

- 일정 마스터 화면 안에 `AI 제안 패널`을 추가한다.
- 패널은 `대화 영역`과 `제안 카드 영역`을 함께 가진다.
- 사용자가 현재 보고 있는 일정표를 벗어나지 않고 제안을 검토하도록 한다.

권장 UI 요소:

- `AI 제안 패널 열기` 액션 버튼
- 현재 선택 범위 표시
- 대화 스레드 영역
- 프롬프트 입력창
- 제안 카드 목록
- `Before -> After` 비교 표시
- 제안 사유 및 예상 영향 표시
- `사용`, `거절`, `다른 안 보기`, `전체 적용`, `모두 닫기` 액션

### 6.2 제안 카드 구조

제안(Proposal):

- 카드 헤더: 대상 공정명, 범위, 우선순위
- 변경 본문: `변경 전`, `변경 후`
- 설명 영역: 제안 이유, 주의사항, 기대 효과
- 액션 영역: `사용`, `거절`, `대화에 반영`, `같은 조건으로 재제안`

### 6.3 시각적 원칙

제안(Proposal):

- 실제 반영 전에는 원본 데이터와 제안 데이터를 명확히 구분한다.
- 승인되지 않은 제안은 원본을 덮어쓰지 않도록 표현한다.
- 셀 단위 강조, 배지, 하이라이트 등으로 제안 존재를 명확히 알린다.
- 패널에서 특정 카드를 hover 또는 선택하면 표의 대상 행/셀도 연동 강조된다.

## 7. 시스템 개념 모델

### 7.1 대화 스레드

제안(Proposal):

- 한 번의 패널 사용 흐름은 `스레드(thread)` 단위로 관리한다.
- 스레드는 사용자 메시지, AI 메시지, 제안 카드, 사용자 결정 이벤트를 포함한다.
- 스레드는 저장 전 임시 상태와 저장 후 이력 상태를 구분할 수 있어야 한다.

### 7.2 제안 카드

제안(Proposal):

- 제안 카드는 단순 텍스트가 아니라 구조화된 데이터여야 한다.
- 최소 필드 예시:
  - `proposal_id`
  - `target_item_id`
  - `target_field`
  - `before_value`
  - `after_value`
  - `reason`
  - `impact_summary`
  - `status(pending/applied/rejected/revised)`

### 7.3 사용자 결정 이벤트

제안(Proposal):

- 사용자의 `사용`, `거절`, `재요청`은 별도 이벤트로 기록한다.
- 실제 데이터 반영 시 어떤 제안을 언제 채택했는지 추적 가능해야 한다.

## 8. MVP 범위

### 8.1 포함 범위(In)

제안(Proposal):

- 선택한 행 또는 대공종 대상 대화형 제안 생성
- 패널 내 자연어 요청 입력
- AI 설명 메시지 + 구조화된 제안 카드 반환
- 제한된 필드에 대한 변경 후보 제시
- 제안 사유 표시
- 개별 `사용`/`거절`
- 전체 적용
- `이 제안 다시 받기` 수준의 단순 후속 요청
- 적용 후 기존 편집/저장 흐름과 연결

### 8.2 제외 범위(Out)

제안(Proposal):

- 전체 공정표 자동 재작성
- 링크/선후행 자동 수정
- 사용자 승인 없는 자동 저장
- 공정표와 무관한 범용 지식형 챗봇
- 프로젝트 전체 대상 완전자율 최적화
- 다중 사용자의 실시간 공동편집 충돌 해결

## 9. 기대 효과

제안(Proposal):

- 공정표 검토 속도 향상
- 초안 작성자의 입력 편차 완화
- 변경 이유의 명문화
- 일정 조정 과정의 가시성 확보
- AI 기능에 대한 실무 사용자 신뢰 향상
- `대화하면서 조정`하되 `적용은 사람이 승인`하는 안전한 사용성 확보

특히 본 시스템에서는 `제안을 왜 했는지 설명할 수 있는가`와 `사용자가 수정 반영 여부를 직접 결정할 수 있는가`가 기능 만족도에 직접적인 영향을 준다.

## 10. 리스크 및 대응 방향

### 10.1 주요 리스크

제안(Proposal):

- 도메인 맥락을 무시한 부정확한 제안
- AI 제안을 정답처럼 받아들이는 사용자 오해
- 변경 이력 부재 시 책임 추적 어려움
- 대규모 일정표에서 응답 지연 발생
- 외부 AI 서비스 사용 시 데이터 노출 우려
- 후속 대화가 길어질수록 맥락이 누적되어 제안 품질이 흔들릴 가능성

### 10.2 대응 방향

제안(Proposal):

- 제안 범위를 제한된 필드부터 시작한다.
- 제안 사유와 근거를 반드시 노출한다.
- 사용자 승인 절차를 필수로 둔다.
- 반영 이력과 수락 여부를 기록한다.
- AI는 계산 엔진이 아니라 제안 엔진으로 위치시킨다.
- 스레드 길이, 재제안 횟수, 컨텍스트 범위를 운영 정책으로 제한한다.

## 11. 단계별 도입 방향

### 11.1 1단계

제안(Proposal):

- 선택 범위에 대한 대화형 요청 입력
- AI 설명 메시지와 제안 카드 표시
- 변경 전/후 비교 표시
- 개별 사용/거절

### 11.2 2단계

제안(Proposal):

- 후속 질문 기반 재제안
- 전체 적용
- 적용 후 되돌리기 흐름 연동
- 패널 카드와 테이블 하이라이트 연동

### 11.3 3단계

제안(Proposal):

- 제안 이력 및 반영 기록 관리
- 거절 패턴 기반 재제안 품질 개선
- 운영 피드백 수집 및 제안 정책 튜닝

## 12. 오픈 이슈

오픈 이슈(Open Issue):

- 제안 카드 단위 적용 시 다수 필드 변경을 하나의 트랜잭션으로 볼지 여부
- 후속 대화 맥락을 프론트 상태로만 유지할지 서버 세션으로 저장할지 여부
- 사용자 거절 이력을 다음 제안 생성에서 얼마나 강하게 반영할지 여부
- 제안 결과를 저장 전 임시 상태로만 둘지, 프로젝트 이력으로 남길지 여부
- AI 응답 실패 시 기존 규칙 기반 추천과 어떻게 폴백할지 여부
