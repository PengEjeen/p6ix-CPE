# AI 수정 제안 패널 구현 계획

## 1. 목표/배경

- 일정 마스터 화면에서 기존 `기간 조정` 기능을 `대화형 AI 제안 패널` 형태로 재구성한다.
- 사용자는 우측 패널에서 요청을 입력하고, AI가 생성한 수정 제안 카드를 검토한 뒤 `사용` 또는 `거절`을 결정할 수 있어야 한다.
- 초기 구현은 기존 프론트 내 일정 조정 로직을 재사용하는 MVP로 한정한다.

## 2. 범위(In) / 범위(Out)

In:

- 일정 마스터 화면 우측 패널 UI 추가
- 사용자 요청 입력창과 대화 메시지 표시
- 기존 AI 조정 실행 결과를 제안 카드로 변환
- 제안 카드별 `사용` / `거절`
- 전체 적용
- 간트/테이블 화면에서 동일 패널 표시

Out:

- 신규 백엔드 AI 제안 API 설계 및 구현
- 서버 저장형 대화 스레드
- 다중 사용자 동시 편집 충돌 처리
- 링크/선후행 자동 수정

## 3. 작업 단위

### 3.1 패널 상태 모델 정리

- 기존 `useAIScheduleOptimizer`에 대화 메시지와 제안 카드 상태를 추가한다.
- 사용자 입력 문장에서 목표 공기 수치를 보조 추출하는 최소 규칙을 추가한다.

### 3.2 제안 카드 생성 로직 추가

- 기존 미리보기 결과와 원본 데이터를 비교해 행 단위 제안 카드를 만든다.
- 카드에는 대상 공정, 변경 필드, 사유, 예상 효과를 담는다.

### 3.3 패널 UI 구현

- 우측 패널에 대화 스레드, 제안 카드 목록, 입력창, 전체 적용 액션을 배치한다.
- 카드 hover/상태 표현은 기존 톤을 유지한다.

### 3.4 부분 적용 흐름 연결

- 카드별 `사용` 시 해당 변경만 스토어에 반영한다.
- `거절` 시 pending 상태에서 제외한다.
- `전체 적용`은 기존 미리보기 전체 반영 흐름을 유지한다.

### 3.5 화면 통합

- `ScheduleMasterList`에서 테이블/간트 공통 레이아웃으로 패널을 표시한다.
- 헤더에 패널 토글 액션을 추가한다.

## 4. 검증 방법

- 코드 변경 diff 자체 검토
- `cd frontend && npm run build`
- 수동 검증:
  - 패널 열기/닫기
  - 요청 입력 후 제안 카드 생성
  - 개별 사용/거절
  - 전체 적용
  - 간트/테이블 전환 시 패널 유지 여부 확인

## 5. 리스크/대응

- 리스크: 기존 AI 조정은 `목표 공기` 기반이라 자유 대화형 기대와 차이가 있을 수 있음
- 대응: MVP에서는 자연어 입력을 `제안 요청 컨텍스트`와 `목표 공기 추출 보조`로 제한한다

- 리스크: 부분 적용 후 미리보기와 현재 데이터의 기준이 어긋날 수 있음
- 대응: 적용된 카드의 before 기준을 즉시 갱신해 남은 제안만 pending으로 유지한다

- 리스크: 테이블 화면 폭이 줄어들 수 있음
- 대응: 패널 너비를 고정 범위로 제한하고 토글 가능하게 둔다

## 6. 롤백 계획

- 패널 관련 신규 컴포넌트와 훅 확장 부분만 되돌린다
- 기존 `기간 조정 -> 로그 -> 전체 적용` 흐름은 유지 가능하도록 변경 범위를 제한한다

## 7. 완료 기준(Definition of Done)

- 일정 마스터 화면에서 AI 패널을 열 수 있다
- 사용자가 요청을 입력하면 대화 메시지와 제안 카드가 표시된다
- 카드별 `사용`과 `거절`이 동작한다
- 전체 적용이 동작한다
- 프론트 빌드가 통과한다
