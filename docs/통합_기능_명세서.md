## 1. 목적과 범위

### 1.1 목적
본 문서는 공기검토플랫폼의 기능 요구사항을 통합 관점에서 정리한 초안이다. 기존 [`공기검토플랫폼_SRS.md`](/home/pengejeen/p6ix-CPE/공기검토플랫폼_SRS.md), [`260306_설계서.md`](/home/pengejeen/p6ix-CPE/260306_설계서.md), 사용자 가이드와 현재 구현 코드를 기준으로 다음 항목을 일관되게 설명한다.

- 프로젝트 생성 및 관리
- 아파트 공기 계산(`APARTMENT`)
- 전체 공기산정(`TOTAL`)
- 가동률 산정
- 생산성 근거 및 표준품셈 관리
- 공정 스케줄/간트/내보내기
- 견적 및 AI 보조 기능

### 1.2 범위
포함 범위:

- React SPA와 Django REST API를 통한 웹 기능
- 프로젝트 단위 데이터 생성, 조회, 수정, 삭제(논리 삭제 포함)
- 기상/공휴일 기반 가동률 계산
- 표준 생산성, CIP, 기성말뚝, 현장타설말뚝 데이터 관리
- 스케줄 JSON 저장, 엑셀/DOCX 내보내기

제외 범위:

- ERP/PMIS 등 외부 업무 시스템 연동
- 실시간 현장 제어 자동화
- 운영 인프라 상세 구성, 모니터링 체계, 백업 자동화 구현 세부

### 1.3 문서 해석 원칙

- `사실(Fact)`: 현재 코드에서 확인된 동작
- `가정(Assumption)`: 기존 문서에는 있으나 코드에서 완전 검증되지 않은 항목
- `오픈 이슈(Open Issue)`: 구현/정책 정리가 더 필요한 항목

## 3. 핵심 기능

### 3.1 프로젝트 관리

사실(Fact):

- 프로젝트는 `APARTMENT`와 `TOTAL` 두 유형을 가진다.
- 생성 시 `start_date`가 없으면 생성일 기준으로 자동 설정된다.
- 삭제는 `is_delete=True` 기반 논리 삭제다.
- 프로젝트 생성 시 유형별 초기 데이터가 자동 생성된다.

주요 기능:

| 기능 ID | 기능명 | 설명 |
| --- | --- | --- |
| PRJ-01 | 프로젝트 생성 | 제목, 설명, 유형을 입력해 프로젝트를 생성한다. |
| PRJ-02 | 프로젝트 목록 조회 | 로그인 사용자 기준 프로젝트를 조회한다. |
| PRJ-03 | 프로젝트 상세/수정 | 기본 정보와 시작일을 조회/수정한다. |
| PRJ-04 | 프로젝트 삭제 | 논리 삭제 후 기본 목록에서 제외한다. |
| PRJ-05 | 초기 데이터 생성 | 유형별 계산 입력, 스케줄, 생산성/기준 데이터를 자동 생성한다. |

### 3.2 아파트 공기 계산

사실(Fact):

- 프로젝트 생성 시 `ConstructionOverview`, `WorkCondition`, `PreparationPeriod`, `EarthworkInput`, `FrameWorkInput`, `Quotation`이 생성된다.
- 적용기준은 관리자 템플릿이 있으면 복제하고, 없으면 빈 기본값으로 생성한다.
- 가동률 카테고리도 프로젝트 생성 시 함께 초기화된다.

주요 입력 영역:

| 기능 ID | 기능명 | 설명 |
| --- | --- | --- |
| APT-01 | 공사개요 입력 | 발주처, 건물용도, 공사형식, 위치, 대지/층수/세대수/연면적 등을 관리한다. |
| APT-02 | 근무조건/가동률 입력 | 지역, 데이터 기간, 토공/골조 주간 작업일, 직접 입력 가동률 사용 여부를 관리한다. |
| APT-03 | 준비/정리/마감 입력 | 준비기간, 세대수 구간, 마감공사 직접 입력 여부를 관리한다. |
| APT-04 | 토공사 입력 | 굴착공법, 흙막이공법, 심도, 투입조, 지보공 조건 등을 관리한다. |
| APT-05 | 골조공사 입력 | 골조 조건 및 산정 입력값을 관리한다. |
| APT-06 | 견적 조회/수정 | 기간 항목별 값과 비고를 확인/수정한다. |
| APT-07 | AI 견적 분석 | 입력/기준/견적 데이터를 바탕으로 AI 분석을 비동기 요청한다. |

### 3.3 전체 공기산정

사실(Fact):

- 프로젝트 생성 시 `WorkCondition`이 생성되고 표준 생산성/생산성 근거 템플릿이 프로젝트용으로 복제된다.
- 스케줄은 프로젝트당 1개의 JSON 컨테이너로 관리된다.
- 스케줄 기본 데이터는 프로젝트 생성 시 바로 생성된다.

주요 기능:

| 기능 ID | 기능명 | 설명 |
| --- | --- | --- |
| TOT-01 | 가동률 관리 | 대공종별 기후 기준, 주간 작업일, 공공/민간 구분, 산출 결과를 관리한다. |
| TOT-02 | 표준품셈 관리 | 공종/세부공종/규격/단위/작업조/작업량 기준을 관리한다. |
| TOT-03 | CIP 생산성 근거 | 기준 데이터와 결과 데이터를 조회/수정한다. |
| TOT-04 | 기성말뚝 생산성 근거 | 기준 데이터와 결과 데이터를 조회/수정한다. |
| TOT-05 | 현장타설말뚝 생산성 근거 | 기준 데이터와 결과 데이터를 조회/수정한다. |
| TOT-06 | 공사기간 산정 스케줄 | 항목/링크/부공종을 포함한 스케줄 JSON을 저장하고 편집한다. |
| TOT-07 | 내보내기 | 엑셀(`xlsx`)과 보고서(`docx`)를 다운로드한다. |

### 3.4 가동률 산정

사실(Fact):

- 기상 데이터는 `WeatherDailyRecord`, 공휴일 데이터는 `PublicHoliday`를 참조한다.
- 계산 기준은 동절기, 혹서기, 강우, 강설, 풍속이며 비시정/미세먼지 필드는 존재하지만 계산 반영 여부는 일부 제한적이다.
- 공휴일 데이터가 대상 연도에 없으면 2025년 데이터를 대체 참조한다.

산출값:

- `working_days`: 작업 가능일
- `climate_days_excl_dup`: 기후불능일(중복 제외)
- `legal_holidays`: 법정공휴일 및 주간 휴일 반영 값
- `operating_rate`: 최종 가동률(%)

### 3.5 스케줄 및 간트

사실(Fact):

- 스케줄 JSON은 `items`, `links`, `sub_tasks` 구조를 지원한다.
- 저장 시 스케줄의 대공종 기준으로 가동률 항목 자동 생성/갱신이 연동된다.
- `initialize_default`, `export-excel`, `export-report` 커스텀 액션이 존재한다.

지원 데이터 예:

```json
{
  "items": [
    {
      "id": "earth-1",
      "main_category": "토공사",
      "process": "토사운반",
      "work_type": "토사운반",
      "quantity": 1000,
      "unit": "m3",
      "productivity": 50,
      "crew_size": 5,
      "daily_production": 250,
      "working_days": 4.0,
      "calendar_days": 5.3,
      "calendar_months": 0.18
    }
  ],
  "links": [],
  "sub_tasks": []
}
```

## 4. 주요 API 흐름

### 4.1 프로젝트 생성 흐름

1. 클라이언트가 `POST /api/cpe/project/create/` 호출
2. 서버가 프로젝트 기본 정보를 저장
3. `calc_type`에 따라 아파트 입력 모델 또는 전체 공기산정 템플릿 데이터를 생성
4. 공통으로 기본 스케줄 컨테이너를 생성
5. 응답으로 생성된 프로젝트 정보를 반환

관련 엔드포인트:

- `GET /api/cpe/project/`
- `POST /api/cpe/project/create/`
- `GET /api/cpe/project/{project_id}/`
- `PATCH|PUT /api/cpe/project/{project_id}/update/`
- `DELETE /api/cpe/project/{project_id}/delete/`

### 4.2 가동률 조회/저장 흐름

1. 클라이언트가 `GET /api/cpe/work-schedule-weights/{project_id}/`로 대공종별 설정과 결과를 조회
2. 사용자가 기준값과 설정값을 수정
3. 클라이언트가 `PUT /api/cpe/work-schedule-weights/{project_id}/update/` 호출
4. 서버가 기상/공휴일 데이터 기준으로 가동률을 재계산하고 저장
5. 결과 배열을 반환

관련 엔드포인트:

- `GET /api/cpe/work-schedule-weights/`
- `GET /api/cpe/work-schedule-weights/{project_id}/`
- `PUT /api/cpe/work-schedule-weights/{project_id}/update/`

### 4.3 아파트 계산 입력 흐름

프로젝트별 입력 리소스를 개별 조회/수정한다.

- `GET /api/cpe/calc/construction-overview/{project_id}/`
- `PUT|PATCH /api/cpe/calc/construction-overview/{project_id}/update/`
- `GET /api/cpe/calc/work-condition/{project_id}/`
- `PUT|PATCH /api/cpe/calc/work-condition/{project_id}/update/`
- `GET /api/cpe/calc/preparation-period/{project_id}/`
- `PUT|PATCH /api/cpe/calc/preparation-period/{project_id}/update/`
- `GET /api/cpe/calc/earthwork-input/{project_id}/`
- `PUT|PATCH /api/cpe/calc/earthwork-input/{project_id}/update/`
- `GET /api/cpe/calc/framework-input/{project_id}/`
- `PUT|PATCH /api/cpe/calc/framework-input/{project_id}/update/`

가정:

- 실제 HTTP 메서드 사용은 뷰 구현에 따라 `PATCH`/`PUT` 혼용 가능성이 있어 API 테스트 정리가 필요하다.

### 4.4 스케줄 저장/초기화/내보내기 흐름

1. 클라이언트가 `GET /api/cpe-all/schedule-item/?project_id={project_id}`로 스케줄 컨테이너를 조회
2. 없거나 초기화가 필요하면 `POST /api/cpe-all/schedule-item/initialize_default/`
3. 사용자가 테이블/간트에서 편집 후 `PUT|PATCH /api/cpe-all/schedule-item/{id}/`
4. 필요 시 `GET /api/cpe-all/schedule-item/export-excel/?project_id={project_id}`
5. 필요 시 `GET /api/cpe-all/schedule-item/export-report/?project_id={project_id}`

관련 응답 파일:

- 엑셀: `공사기간_산정_기준_{프로젝트명}.xlsx`
- 보고서: `공사기간_산정_보고서_{프로젝트명}.docx`

### 4.5 AI 견적 분석 흐름

1. 클라이언트가 `POST /api/cpe/quotation/{project_id}/ai_update/` 호출
2. 서버가 기존 `ai_response`를 초기화
3. 견적/기준/입력 데이터를 수집
4. AI 작업을 큐에 등록하고 `202 Accepted` 응답 반환
5. 백그라운드 실행 완료 후 `Quotation.ai_response`에 결과 저장
6. 클라이언트가 견적 상세 재조회로 결과 확인

오픈 이슈:

- 현재 구현상 `ai_update` 엔드포인트에 인증 데코레이터가 비활성화되어 있어 보안 정책 정리가 필요하다.

## 5. 데이터 모델 개요

### 5.1 핵심 엔터티

| 엔터티 | 설명 | 주요 필드 |
| --- | --- | --- |
| `Project` | 프로젝트 중심 엔터티 | `title`, `description`, `calc_type`, `start_date`, `is_delete` |
| `ConstructionOverview` | 아파트 공기 계산 기본 정보 | 발주처, 위치, 층수, 세대수, 연면적 |
| `WorkCondition` | 근무조건 및 가동률 입력 | `region`, `data_years`, `earthwork_type`, `framework_type` |
| `PreparationPeriod` | 준비/정리/마감 기간 입력 | 준비기간, 세대수 구간, 마감 직접입력 여부 |
| `EarthworkInput` | 토공사 입력 | 굴착공법, 흙막이 공법, 심도, 투입조 |
| `FrameWorkInput` | 골조공사 입력 | 골조 계산 입력값 |
| `WorkScheduleWeight` | 대공종별 가동률 | 기후 기준, 주간 작업일, 공공/민간, 산출값 |
| `Quotation` | 아파트 기간 견적서 | 준비/토공/골조/마감 기간, 비고, `ai_response` |
| `ConstructionProductivity` | 표준 생산성 데이터 | 공종 분류, 작업조, 생산성 기준, 작업량 |
| `CIP*`, `Pile*`, `BoredPile*` | 생산성 근거/결과/표준 | 공법별 입력/기준/결과 |
| `ConstructionScheduleItem` | 스케줄 JSON 컨테이너 | `project`, `data` |
| `WeatherStation` | 기상 관측소 | `station_id`, `name` |
| `WeatherDailyRecord` | 일별 기상 데이터 | 날짜, 기온, 풍속, 강수량, 적설 등 |
| `PublicHoliday` | 공휴일 데이터 | 날짜, 공휴일명, 공공/민간 적용 여부 |

### 5.2 관계 요약

- 사용자 1명은 여러 프로젝트를 가질 수 있다.
- 프로젝트 1건은 계산 입력, 가동률, 견적, 스케줄, 생산성 데이터의 기준 단위다.
- `ConstructionScheduleItem`은 현재 프로젝트당 단일 컨테이너 사용을 전제로 한다.
- 가동률은 스케줄 대공종과 매핑되는 `main_category`를 기준으로 저장된다.

## 6. 비기능 요구사항

### 6.1 성능

- 일반 사무용 PC와 일반 인터넷 환경에서 사용 가능해야 한다.
- 프로젝트 단위 CRUD와 계산성 API는 사용자 체감상 지연 없이 응답해야 한다.
- 스케줄 내보내기와 AI 분석은 상대적으로 오래 걸릴 수 있으며, 파일 생성/비동기 응답을 허용한다.

### 6.2 신뢰성

- 프로젝트 생성 시 필요한 초기 데이터는 트랜잭션 범위에서 함께 생성되어야 한다.
- 동일 프로젝트 데이터는 재조회 시 일관된 값으로 반환되어야 한다.
- 스케줄 저장 실패 시 부분 저장보다 오류 반환을 우선한다.

### 6.3 보안

- 보호 API는 인증 사용자만 접근 가능해야 한다.
- 사용자는 본인 소유 프로젝트만 조회/수정 가능해야 한다.
- 외부 AI/SSO 연동 정보와 비밀값은 환경변수로 관리해야 한다.

### 6.4 유지보수성

- 프론트는 `pages/components/api/stores/hooks/utils`, 백엔드는 `models/serializers/views/urls` 분리 구조를 유지한다.
- 무관한 리팩터링이나 대규모 구조 변경 없이 요청 범위 내 수정이 원칙이다.

### 6.5 운영/배포

- 운영 DB는 PostgreSQL, 로컬 개발은 SQLite fallback을 허용한다.
- Docker 멀티스테이지 기반으로 프론트 빌드 산출물과 Django 런타임을 배포한다.

## 7. 에러 및 예외 처리

### 7.1 공통 원칙

- API는 HTTP 상태코드와 JSON 오류 메시지를 반환한다.
- 파일 다운로드 API는 내부 예외 발생 시 `500`과 오류 메시지를 반환한다.

### 7.2 대표 예외

| 상황 | 상태코드 | 현재 동작 |
| --- | --- | --- |
| 인증 없이 보호 API 접근 | `401` 또는 권한 오류 | DRF 인증/권한 정책에 따름 |
| `project_id` 누락 | `400` | `project_id is required` 오류 반환 |
| 스케줄 데이터 없음 | `404` | `schedule data not found` 반환 |
| 스케줄 데이터 형식 오류 | `400` | `invalid schedule data` 반환 |
| serializer 검증 실패 | `400` | 필드별 validation 오류 반환 |
| 파일 내보내기 내부 예외 | `500` | `export-excel failed`, `export-report failed` 반환 |

### 7.3 데이터 예외

- 대상 연도 공휴일 데이터가 없으면 2025년 데이터를 대체 사용한다.
- 스케줄 컨테이너가 중복 생성된 경우 초기화 API에서 기존 다건을 정리한다.
- 스케줄 대공종이 변경되면 관련 가동률 항목 생성/동기화가 필요하다.

## 8. 수용 기준

### 8.1 프로젝트 관리

- 사용자는 프로젝트명과 유형을 입력해 프로젝트를 생성할 수 있어야 한다.
- 생성 직후 유형별 기본 데이터와 기본 스케줄이 존재해야 한다.
- 삭제된 프로젝트는 기본 목록 조회에서 제외되어야 한다.

### 8.2 아파트 공기 계산

- 사용자는 공사개요, 근무조건, 준비/정리기간, 토공사, 골조공사 입력값을 저장 후 재조회할 수 있어야 한다.
- 적용기준이 존재할 경우 프로젝트 생성 시 복제되어야 한다.
- 견적 데이터는 프로젝트별로 조회/수정 가능해야 한다.

### 8.3 가동률

- 사용자는 프로젝트별 대공종 가동률 목록을 조회할 수 있어야 한다.
- 주간 작업일, 지역, 연수, 기후 기준 변경 후 저장 시 산출값이 갱신되어야 한다.
- 공휴일과 기후 데이터가 존재할 경우 결과가 숫자값으로 반환되어야 한다.

### 8.4 전체 공기산정

- 프로젝트 생성 시 표준 생산성 및 생산성 근거 템플릿이 프로젝트용으로 생성되어야 한다.
- 사용자는 스케줄 JSON을 저장하고 재조회했을 때 동일 구조를 확인할 수 있어야 한다.
- 엑셀과 보고서 내보내기 호출 시 다운로드 가능한 파일 응답을 받아야 한다.

### 8.5 AI 기능

- AI 견적 분석 요청 시 즉시 `202 Accepted`가 반환되어야 한다.
- 분석 완료 후 견적 상세 재조회 시 `ai_response`가 채워져야 한다.

## 9. 오픈 이슈

1. `quotation/{project_id}/ai_update/`는 현재 인증 데코레이터가 비활성화되어 있다. 실제 운영 권한 정책을 확정해야 한다.
2. `WorkScheduleWeight`의 `visibility_threshold`, `dust_alert_level`은 모델에 존재하지만 계산 로직 반영 범위가 문서와 코드 사이에 완전히 일치하는지 추가 검증이 필요하다.
3. 스케줄 API의 요청 형식은 배열 단독과 `{items, links, sub_tasks}` 컨테이너를 모두 허용하는 흔적이 있다. 단일 표준 계약을 확정해야 한다.
4. 일부 업데이트 API의 메서드(`PUT`/`PATCH`)와 프론트 실제 호출 방식에 대한 계약 문서가 부족하다.
5. 관리자 기준 데이터(`PreparationWork`, `Earthwork`, `FrameWork`)의 운영 입력 주체와 변경 승인 절차가 문서화되어 있지 않다.
6. AI 분석 큐의 실행 보장 방식, 실패 재시도 정책, 타임아웃 정책이 문서화되어 있지 않다.
7. 프로젝트별 스케줄 컨테이너는 사실상 단일 행 전제를 사용하나, DB 차원의 유니크 제약 여부는 추가 점검이 필요하다.
