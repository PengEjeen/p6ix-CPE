## 1. 개요

### 1.1 목적
본 문서는 공기검토플랫폼의 현재 구현을 기준으로 시스템 구조, 모듈 책임, 데이터 구조, 인터페이스 및 배포 방식을 정의한다.  
개발/운영/테스트 팀이 동일한 설계 관점을 공유하도록 하는 것을 목적으로 한다.

### 1.2 범위
- 웹 애플리케이션 전체(프론트엔드 SPA + Django REST API)
- 공기 계산(APARTMENT) 및 전체 공기산정(TOTAL) 도메인
- 생산성 기준 관리(CIP/기성말뚝/현장타설말뚝/표준품셈)
- 공정 스케줄(JSON 컨테이너), 간트 편집, 엑셀/보고서 내보내기
- 가동률 산정(기상/공휴일 반영)
- AI 요약 및 견적 분석 연계

### 1.3 설계 원칙
- 프로젝트 중심 데이터 분리(`Project` FK 기반)
- REST API + Keycloak(OIDC) 공통 로그인 + Django Session 기반의 명확한 경계
- 프론트 상태(store)와 서버 영속 데이터 분리
- 프로젝트 생성 시 기본 데이터 자동 초기화
- 스케줄은 단일 JSON 컨테이너 방식으로 저장/조회

## 2. 시스템 소개

### 2.1 시스템 개요
시스템은 React(Vite) 프론트엔드와 Django REST 백엔드로 구성된다.  
사용자는 로그인 후 프로젝트를 생성하고, 유형(APARTMENT/TOTAL)에 따라 입력/산정/검토/내보내기 기능을 수행한다.

### 2.2 논리 구성도
```text
[Browser (React SPA)]
        |
        | HTTPS/HTTP + JSON, OIDC(Keycloak) + Django Session
        v
[Nginx Reverse Proxy]
   |                 \
   | /api/*           \ static files (frontend dist)
   v
[Django + DRF + Gunicorn]
   |      |      |      \
   |      |      |       \ AI (Gemini API)
   |      |      |
   |      |      +-- cpe_all_module (TOTAL)
   |      +--------- cpe_module (project/calc/criteria/operating_rate/quotation)
   +---------------- user / operatio
        |
        v
[PostgreSQL(or SQLite dev)]
```

### 2.3 기술 스택
| 레이어 | 기술 |
| --- | --- |
| Frontend | React 19, Vite 5, React Router, axios, zustand, zundo, dnd-kit |
| Backend | Django 5, Django REST Framework, django-environ, SimpleJWT(legacy fallback) |
| Auth/SSO | Keycloak (OIDC Authorization Code + PKCE) |
| Data | PostgreSQL(운영), SQLite(개발 fallback) |
| Export | xlsxwriter, openpyxl, python-docx, Pillow |
| AI | google-genai (Gemini) |
| Deploy | Docker, Gunicorn, Nginx |

## 3. 아키텍처 설계

### 3.1 백엔드 앱 구조

### 3.2 프론트엔드 구조

### 3.3 핵심 시퀀스

#### 3.3.1 프로젝트 생성 시퀀스
1. 클라이언트가 `POST /api/cpe/project/create/` 호출
2. `ProjectSerializer.create()`에서 `calc_type` 분기
3. `APARTMENT`:
   - `ConstructionOverview`, `WorkCondition`, `PreparationPeriod`, `EarthworkInput`, `FrameWorkInput` 생성
   - `WorkScheduleWeight` 기본 카테고리 생성 및 가동률 계산
   - admin 기준(`PreparationWork`, `Earthwork`, `FrameWork`) 복제
   - `Quotation` 생성
4. `TOTAL`:
   - `WorkCondition` 생성
   - 표준 템플릿(project=null)에서 생산성/기준 데이터 복제
   - `CIPResult`, `PileResult`, `BoredPileResult` 생성
5. 공통:
   - `ConstructionScheduleItem` 기본 JSON 생성

#### 3.3.2 스케줄 저장 시 가동률 자동 생성
1. 클라이언트가 스케줄 JSON 저장(PATCH)
2. `ConstructionScheduleItem` 저장 완료 후 post-save signal 실행
3. 스케줄 `main_category`(+조건부 process 키) 추출
4. 없는 `WorkScheduleWeight` 자동 생성
5. 기본 설정(서울/10년/6일)으로 가동률 재계산

#### 3.3.3 AI 견적 분석 시퀀스
1. 클라이언트가 `POST /api/cpe/quotation/{project_id}/ai_update/` 호출
2. 서버는 즉시 큐 등록 후 `202 Accepted` 반환
3. 백그라운드 worker가 기준/입력/견적 데이터를 수집
4. Gemini 프롬프트 생성 후 응답 저장(`Quotation.ai_response`)
5. 클라이언트 재조회 시 분석 결과 확인

## 4. 모듈 및 구성요소

### 4.1 사용자/인증 모듈 (`user`)
#### 4.1.1 설명
- 사용자 계정은 `AbstractUser` 확장 모델 사용(UUID PK)
- Keycloak Authorization Code 기반 로그인 후 Django Session 생성
- Google OAuth 및 로컬 JWT 엔드포인트는 legacy fallback으로만 유지

#### 4.1.2 인터페이스
| Method | Path | 설명 |
| --- | --- | --- |
| POST | `/api/users/register/` | 회원가입 |
| GET | `/api/sso/login/` | Keycloak authorize endpoint 리다이렉트 시작 |
| GET | `/api/sso/callback/` | code/state 검증, token 교환, 로컬 유저 동기화, Django session 로그인 |
| GET | `/api/sso/session/` | 현재 세션 인증 사용자 조회 |
| POST | `/api/users/logout/` | 로그아웃 |
| GET/PUT | `/api/users/profile/` | 프로필 조회/수정 |
| PUT | `/api/users/password_change/` | 비밀번호 변경 |
| POST | `/api/users/login/` | (legacy) 로컬 로그인/JWT |
| POST | `/api/users/token/refresh/` | (legacy) Access 재발급 |
| POST | `/api/users/google/login/` | (legacy) Google 로그인 |

#### 4.1.3 동작
- `/login` 진입 시 앱은 `/api/sso/login/`으로 리다이렉트
- 콜백에서 `code/state` 검증 후 Keycloak token endpoint 호출
- ID token/userinfo의 `sub`를 로컬 사용자 `keycloak_sub`와 매핑
- 최초 로그인 시 자동 생성 또는 pending/deny 정책(`SSO_NEW_USER_POLICY`) 적용
- 성공 시 `django.contrib.auth.login()`으로 세션 생성 후 `next` 경로로 복귀

## 5. 데이터 설계

### 5.1 핵심 엔터티
| 엔터티 | 관계 | 설명 |
| --- | --- | --- |
| `User` | 1:N `Project` | 사용자 |
| `Project` | 중심 엔터티 | 유형(`APARTMENT`/`TOTAL`), 시작일, soft delete |
| `ConstructionOverview` 등 calc 입력 | N:1 `Project` | 아파트 계산 입력 |
| `PreparationWork/Earthwork/FrameWork` | N:1 `Project` | 적용기준 |
| `WorkScheduleWeight` | N:1 `Project` | 대공종별 가동률(유니크 제약) |
| `Quotation` | N:1 `Project` | 견적 항목 + AI 응답 |
| `ConstructionProductivity` | N:1 `Project` | 표준품셈/생산성 |
| `CIP/Pile/Bored*` Basis/Result/Standard | N:1 `Project`(basis/result) | 생산성 근거/결과/표준 |
| `ConstructionScheduleItem` | N:1 `Project` | 스케줄 JSON 컨테이너 |
| `WeatherDailyRecord`, `PublicHoliday`, `WeatherStation` | 독립 참조 | 가동률 계산 참조 데이터 |

### 5.2 스케줄 JSON 구조
```json
{
  "items": [],
  "links": [],
  "sub_tasks": []
}
```
- `items`: 공정 행(수량/생산성/인원/달력일/병행정보 등)
- `links`: 선후행(FS/SS/FF/SF + lag)
- `sub_tasks`: 부공종 바 정보

### 5.3 상태관리 설계(프론트)
- `scheduleStore`에서 `items/links/subTasks/operatingRates/workDayType` 관리
- `zundo`로 undo/redo(히스토리 제한 50)
- 로컬 스냅샷 별도 관리(최대 20)

## 6. 인터페이스 설계

### 6.1 인증/권한
- 기본 DRF 권한: `IsAuthenticated`
- 기본 인증 클래스: `SessionAuthentication`
- 필요 시 `LEGACY_JWT_AUTH_ENABLED=true`로 JWT fallback 활성화 가능
- 공통 로그인(IdP): Keycloak(OIDC Authorization Code + PKCE)
- 예외적으로 일부 엔드포인트는 별도 권한 처리 필요(운영 보안 점검 항목)

### 6.2 데이터 포맷
- 요청/응답 기본: `application/json`
- 파일 응답: Excel/DOCX 바이너리(blob)
- 목록 응답: 배열 또는 `{ "results": [...] }` 혼재(클라이언트에서 호환 처리)

### 6.3 오류 처리
- HTTP 상태코드 + JSON 오류 메시지 반환
- 프론트는 API 레이어에서 예외 throw 후 페이지에서 toast/alert 처리

## 7. 배포 및 운영 설계

### 7.1 컨테이너 토폴로지
- `backend` 컨테이너: Django + Gunicorn
- `nginx` 컨테이너: 정적 파일 서빙 + `/api/*` reverse proxy
- 공유 볼륨:
  - `django-static` -> Django static
  - `frontend-dist` -> 빌드된 SPA 정적 파일

### 7.2 실행 흐름
1. 이미지 빌드 시 frontend `npm run build`
2. backend entrypoint에서 `collectstatic` 수행
3. frontend dist를 공유 볼륨으로 복사
4. nginx가 SPA/정적 제공, API는 backend로 프록시

### 7.3 환경설정
| 변수/설정 | 목적 |
| --- | --- |
| `SECRET_KEY` | Django/토큰 서명 키 |
| `DB_*` | PostgreSQL 연결정보 |
| `GEMINI_API_KEY` | AI 기능 활성화 |
| `VITE_API_BASE` | 프론트 API 베이스 경로 |
| `VITE_PERSIST_LOGIN` | 테스트 환경 로그인 유지 동작 |
| `KEYCLOAK_*` | 백엔드 Keycloak 검증 설정(server/realm/client/audience) |
| `VITE_KEYCLOAK_*` | 프론트 Keycloak 로그인 설정(url/realm/client/redirect) |
| `VITE_USE_SESSION_AUTH` | 프론트 세션 인증 모드 활성화 |
| `LEGACY_LOCAL_LOGIN_ENABLED` | 로컬 로그인/JWT 발급 fallback 허용 여부 |
| `SSO_NEW_USER_POLICY` | 신규 사용자 처리 정책(`auto`/`pending`/`deny`) |
